# AUDITORÍA COMPLETA — Sports AI Terminal

## 1. RESUMEN EJECUTIVO DEL PROYECTO

**Sistema:** Terminal cuantitativa multi-agente para predicción deportiva de fútbol
**Stack:** FastAPI (Python) + Next.js (React) + PostgreSQL + Redis
**Arquitectura:** 15 agentes especializados en pipeline secuencial con streaming SSE
**APIs:** Statpal Soccer API (primaria), API-Football (fallback legacy)
**LLM:** DeepSeek (default), con soporte para OpenAI/Anthropic/Grok

---

## 2. ARQUITECTURA DEL PIPELINE DE AGENTES

### 2.1 Pipeline Secuencial (15 agentes)
```
NLPAgent → FixtureResolverAgent → ContextAgent → HistoryAgent → LineupAgent 
→ SentimentAgent → EloAgent → OddsAgent → FeatureAgent → PoissonAgent 
→ MLAgent → MonteCarloAgent → MarketEdgeAgent → RiskAgent → SynthesisAgent
```

### 2.2 Problemas Estructurales Identificados

#### CRÍTICO: Dependencia del LLM para datos estructurados
- **Todos los agentes** que generan análisis narrativo usan LLM para interpretar datos numéricos
- Los prompts del sistema (system prompts) tienen peso significativo en las predicciones
- Los JSON de retorno del LLM pueden fallar silently y caer al fallback (poisson_agent.py:121, montecarlo_agent.py:109, elo_agent.py:123, etc.)
- **Riesgo:** Si el LLM malinterpreta datos numéricos, la predicción final se desvía

#### PROBLEMA: Pipeline puramente secuencial
- Cada agente espera al anterior (no hay paralelismo real)
- FixtureResolver → ContextAgent → HistoryAgent → LineupAgent podrían ejecutarse en paralelo
- OddsAgent depende solo de fixture_id (puede ejecutarse en paralelo con HistoryAgent)
- FeatureAgent solo requiere stats (puede ejecutarse después de HistoryAgent)

#### PROBLEMA: Fallback determinista cuando falta datos
- Cuando `history_data_available=False`, se usan valores hardcodeados:
  - `form_score: 50.0` (history_agent.py:198)
  - `attack_rating: 50.0` (history_agent.py:200)
  - `defense_rating: 50.0` (history_agent.py:201)
- Esto sesga el modelo hacia probabilidades del 33%-33%-33%

## 3. AUDITORÍA DETALLADA POR AGENTE

### 3.1 NLPAgent (Agente 1)
**Función:** Parsea lenguaje natural → equipos, liga, fecha

**Problemas:**
- Fallback parser es muy básico (regex-based con separadores hardcodeados)
- Solo reconoce separadores `vs`, `vs.`, `contra`, `v`, `-`
- No reconoce equipos escritos de forma ambigua
- Temperatura 0.1 (casi determinista) pero el parsing JSON puede fallar

**Mejora sugerida:** Integrar con FixtureResolverAgent directamente usando fuzzy matching

---

### 3.2 FixtureResolverAgent (Agente 2)
**Función:** Resuelve fixture real sin asumir home/away del prompt

**Fortalezas:**
- Algoritmo de scoring robusto con múltiples señales (league overlap, date match)
- Prefetch de fixtures para ambos equipos
- Manejo de alternativas y confianza

**Problemas:**
- `MAX_TEAM_CANDIDATES = 6` puede ser limitante para ligas menores
- `NEXT_FIXTURE_WINDOW = 20` días puede perder partidos si la query es para fecha específica
- No usa API de team_info para verificar identidad del equipo
- Cache TTL de team_search no está definido

**Mejora sugerida:** Integrar búsqueda de equipo por ID directamente con get_team()

---

### 3.3 ContextAgent (Agente 3)
**Función:** Contexto competitivo, standings, stats de temporada

**Problemas:**
- Dependencia total del LLM para generar análisis (CONTEXT_SYSTEM_PROMPT)
- No usa home_season_stats ni away_season_stats para calcular métricas directas
- El parsing de standings itera sobre estructuras anidadas sin validación
- Warnings duplicados si home_id o away_id no se resuelven

**CRÍTICO:** 
- Si league_id o season no están en fixture, `get_standings()` y `get_team_statistics()` nunca se llaman
- Los standings se parsean incorrectamente: busca `entry.get("league").get("standings")` pero la estructura real es `league.standings[group][team]`

---

### 3.4 HistoryAgent (Agente 4)
**Función:** Datos históricos, H2H, forma reciente

**Problemas:**
- Pide 20 partidos pero solo usa los últimos 5 para forma (`home_results[:5]`)
- `calculate_form_score()` usa exponencial decay pero solo con 5 datos (poco significativo)
- `calculate_momentum()` requiere 6 partidos pero puede tener menos y retorna 0.0 silenciosamente
- El H2H se parsea desde la perspectiva del home_team pero no hay verificación
- Fallback stats son completamente neutrales (50% para todo)

**PROBLEMA CRÍTICO - xG:**
- El sistema NO usa xG real de la API
- Attack rating se calcula como `avg_scored * 35` (history_agent.py:75)
- Defense rating se calcula como `100 - avg_conceded * 30` (history_agent.py:65)
- Esto es un PROXY MUY RUDIMENTARIO de xG real

**Mejora sugerida:** Integrar datos xG reales de la API si están disponibles

---

### 3.5 LineupAgent (Agente 5)
**Función:** Alineaciones, lesiones, impacto táctico

**Problemas:**
- Dependencia completa del LLM para análisis táctico
- `get_injuries()` sin fixture_id busca TODAS las lesiones del equipo (mucho ruido)
- Injury impact calculation es lineal: `count * 0.05` capped a 0.5 (muy simplista)
- No distingue entre lesionados de titulares vs suplentes
- No considera antigüedad de la lesión ni impacto en el XI

---

### 3.6 SentimentAgent (Agente 6)
**Función:** Análisis de sentimiento y presión

**Problemas:**
- El prompt dice que debe usar "SOLO los datos del prompt" pero los datos son muy limitados
- Dependencia excesiva del LLM para inferir sentimiento
- No tiene acceso a noticias reales ni redes sociales
- El system prompt dice "evita inventar hechos" pero el LLM puede inventar

**Mejora sugerida:** Integrar API de noticias reales o al menos proporcionar más contexto

---

### 3.7 EloAgent (Agente 7)
**Función:** Rating ELO + interpretación

**Problemas:**
- LEAGUE_BASE_ELO hardcodeado para 34 ligas
- K_FACTOR = 32 es estándar pero podría calibrarse
- `_elo_from_form()` combina stats con pesos arbitrarios
- `_adjust_elo_from_results()` usa solo últimos 5 partidos
- No hay actualización de ELO después de cada partido (es estático por predicción)

**PROBLEMA CRÍTICO - ELO Reliability:**
- El sistema NO persiste ELO entre predicciones
- Cada predicción recalcula ELO desde cero
- No hay histórico de ELO del equipo en la base de datos

---

### 3.8 OddsAgent (Agente 8)
**Función:** Cuotas del mercado, probabilidades implícitas

**Problemas:**
- Si no hay fixture_id, retorna sin datos (sin intentar buscarlo)
- No hay manejo de cuotas de más de X bookmakers (podría haber outliers)
- El overround se calcula pero no se usa para ajustar decisiones
- No hay análisis de movement direction (solo magnitud)

---

### 3.9 FeatureAgent (Agente 9)
**Función:** Feature engineering para ML

**Problemas:**
- Solo genera features, no las usa para predicción (el ML model no está entrenado)
- Features son básicas y redundantes con stats ya calculadas
- No hay selección de features ni importancia de features
- `home_advantage` siempre es 1.0 (no se usa)

---

### 3.10 PoissonAgent (Agente 10)
**Función:** Modelo de distribución de goles

**Problemas:**
- Depende de `calculate_expected_goals()` que usa attack/defense ratings proxies
- `league_avg_goals = 2.7` hardcodeado (no es igual para todas las ligas)
- Home advantage de 1.15 (15%) hardcodeado
- No considera variaciones de liga en scoring patterns
- El LLM puede malinterpretar las probabilidades

---

### 3.11 MLAgent (Agente 11)
**Función:** API predictions + cross-validation con LLM

**CRÍTICO - Problema de Arquitectura:**
- El "ML" del nombre es engañoso - no hay modelo de machine learning entrenado
- Solo compara Poisson vs API/market odds
- El fallback usa pesos hardcodeados (60% API, 40% Poisson)
- `get_predictions()` de Statpal siempre retorna None (statpal no tiene endpoint)
- Se usa odds implícitas como proxy de API (ml_agent.py:71-81)

**Mejora sugerida:** Entrenar un modelo real con features histórica

---

### 3.12 MonteCarloAgent (Agente 12)
**Función:** 50K simulaciones Monte Carlo

**Problemas:**
- Usa los mismos λ (lambda) del PoissonAgent
- `most_likely_score` se alinea con el winner general (line 100-109) lo cual puede generar inconsistencia
- LLM interpretation puede contradecir los datos numéricos
- 50K simulaciones está bien pero podría ser 100K para mayor precisión

---

### 3.13 MarketEdgeAgent (Agente 13)
**Función:** Detectar value bets

**Problemas:**
- Aggregation weights hardcodeados: poisson 0.25, ml 0.35, mc 0.25, elo 0.15
- Edge threshold de 5% es arbitrario
- El LLM puede detectar "falsos positivos" pero no hay validación histórica
- No considera:流动性 del mercado, bookmaker reputation, ni sharp vs soft books

**CRÍTICO - Missing Data Blindspots:**
- No considera información de arbitro
- No considera clima
- No considera tiempo de descanso entre partidos
- No considera motivación específica (derbies, partidos grandes)

---

### 3.14 RiskAgent (Agente 14)
**Función:** Kelly Criterion, gestión de stake

**Problemas:**
- Kelly fraction de 0.25 (quarter Kelly) es conservador pero arbitrario
- `kelly_criterion()` puede retornar valores negativos si prob < implied (line 116)
- Confidence score calculation es heurística con pesos hardcodeados
- `recommendation_style` solo tiene "GANARLA" o "ARRIESGARSE" (binario)

**Mejora sugerida:** Kelly fraccional adaptativo basado en confidence

---

### 3.15 SynthesisAgent (Agente 15 - El cerebro final)
**Función:** Executive Summary

**Problemas:**
- Dependencia TOTAL del LLM para el veredicto final
- Si el LLM contradice los datos numéricos, no hay mecanismo de corrección
- No hay ensemble de LLMs para verificar consistencia
- Temperature 0.4 puede generar variabilidad excesiva en resultados similares

---

## 4. PROBLEMAS TRANSVERSALES (TODOS LOS AGENTES)

### 4.1 Manejo de Errores
- Todos los agentes capturan excepciones genéricas
- No hay retry logic para API calls fallidas
- No hay circuit breaker para APIs no disponibles
- Los errores se loggean pero no hay alertamiento

### 4.2 Cache
- TTL de cache dispersos: 900s, 1800s, 3600s, 86400s
- No hay invalidación de cache cuando cambian datos
- El cache de team_search depende del nombre (puede variar)

### 4.3 LLM Dependency
- 13 system prompts diferentes para el mismo LLM
- No hay evaluación de calidad de respuestas del LLM
- No hay versionamiento de prompts
- Costos de LLM no se monitorizan

## 5. FLUJOS DE TRABAJO PROBLEMÁTICOS

### 5.1 Flujo de predicción sin datos
```
Query → NLPAgent → FixtureResolverAgent (falla) → ContextAgent (falla) 
→ HistoryAgent (usa fallback 50/50) → ... → SynthesisAgent (genera resumen con datos invented)
```

### 5.2 Flujo cuando la API no tiene predicciones
- MLAgent usa odds implícitas como proxy
- El sistema reporta "api+poisson" pero en realidad solo tiene Poisson
- La weighted average 60/40 no está validada

### 5.3 Flujo de agregación de probabilidades
- MarketEdgeAgent combina 4 fuentes con pesos hardcodeados
- Si alguna fuente falta, el total_weight se recalcula sin esa fuente
- Los componentes pueden tener probabilidades contradictorias

## 6. DATOS Y MODELOS

### 6.1 Modelo de Datos (Pydantic)
- PredictionResult tiene campos legacy que no se usan
- PredictionRecord en DB es simplificado vs PredictionResult en memoria
- No hay schema evolution para nuevos campos

### 6.2 Base de Datos
- PredictionRecord guarda veredicto pero NO guarda probabilities finales
- No hay tabla de features para entrenamiento futuro
- No hay tabla de resultados reales para comparar con predicciones

### 6.3 Modelos ML
- **EXISTE TrainingService** con XGBoost, Random Forest, Logistic Regression
- **PROBLEMA:** Entrena con DATOS SINTÉTICOS, no datos reales de la API
- El modelo entrenado NO se usa en el pipeline actual
- El pipeline actual solo usa Poisson + comparaciones de odds

### 6.4 Datos de la API
- **Statpal NO proporciona xG** - solo usa marcadores (goals)
- **Statpal NO proporciona match statistics** (posesión, disparos, corners)
- **Statpal solo proporciona:** fixtures, H2H, team_stats (W/D/L/goals), injuries, odds, standings
- **API-Football SÍ tiene Statistics** según su cobertura, pero no se usa como primary

## 7. RECOMENDACIONES PRIORIZADAS

### PRIORIDAD CRÍTICA (Impacto directo en precisión)

1. **Validar Capacidades Reales de Statpal**
   - El cliente Statpal NO implementa endpoint de match statistics
   - Verificar si Statpal tiene `{sport}/matches/{id}/statistics` o similar
   - Si no lo tiene, evaluar migrar a API-Football o complementar

2. **Obtener xG de API o Calcularlo**
   - Statpal no parece proporcionar xG
   - Opciones: (a) Calcular xG desde shots data si la API lo provee, (b) Usar API-Football como fuente
   - TrainingService ya tiene `xg_difference` como feature pero no está implementado

3. **Entrenar Modelo ML Real**
   - Current: TrainingService usa datos sintéticos
   - Ideal: Entrenar modelo con features históricas y resultados reales de la API
   - Necesita: obtener match statistics reales para features

4. **Sistema de Validación (Backtesting)**
   - Actual: No hay forma de evaluar precisión histórica
   - Ideal: Guardar predicciones con resultado real, calcular accuracy
   - Requisito: guardar más datos en PredictionRecord

5. **Agregar Match Statistics al Pipeline**
   - Faltan: posesión, disparos, corners, tarjetas (de la API)
   - Estos datos mejorarían significativamente las predicciones si están disponibles
   - Implementar agente o extensión de HistoryAgent para obtenerlos

### PRIORIDAD ALTA (Mejora de calidad)

6. **Parallelizar Agentes Independientes**
   - HistoryAgent, LineupAgent, OddsAgent pueden ejecutarse en paralelo
   - Reduciría latencia significativamente

7. **Desacoplar Síntesis del LLM**
   - El veredicto final no debería depender 100% del LLM
   - Crear un sistema de reglas/voting para el veredicto

8. **Dynamic ELO con Persistencia**
   - Actualizar ELO después de cada partido real
   - Mantener histórico de ELO en DB

9. **Kelly Criterion Adaptativo**
   - Ajustar Kelly fraction basado en confidence y edge
   - Considerar Kelly completo vs fractional por situación

### PRIORIDAD MEDIA (Estabilidad y Mantenimiento)

10. **Retry Logic y Circuit Breaker**
    - Implementar exponential backoff para API calls
    - Circuit breaker cuando una API falla repetidamente

11. **Monitoreo de Costos LLM**
    - Track tokens por predicción
    - Alertar si costo excede threshold

12. **Cache Invalidation Strategy**
    - Invalidar cache cuando llegan resultados de partidos
    - Fresh data para partidos próximos

## 8. MÉTRICAS DE EVALUACIÓN SUGERIDAS

Para mejorar las predicciones, el sistema debería trackear:

| Métrica | Descripción | Meta |
|---------|-------------|------|
| Accuracy | % de predicciones correctas (1X2) | > 55% |
| Brier Score | Error cuadrático medio de probabilidades | < 0.25 |
| ROI | Return on Investment de apuestas | > 5% |
| Calibration | Prob predicha vs frecuencia real | < 0.05 |
| Edge Realizado | Edge real vs edge predicho | > 80% |

## 9. CONCLUSIÓN

El sistema tiene una **arquitectura robusta** con 15 agentes especializados y usa buenas prácticas (caching, streaming SSE, Redis). Sin embargo, **la mayor debilidad es la dependencia excesiva del LLM** para interpretar datos y generar el veredicto final, combined with **hardcoded heuristics** que代替 actual data-driven models.

**El camino para mejorar precisión:**
1. Reemplazar heuristics con modelos entrenados con datos reales
2. Implementar sistema de backtesting para validar cambios
3. Reducir dependencia del LLM para decisiones finales
4. Agregar más features de entrada (xG real, arbitro, clima)
